# Fraud & Investigation Monitoring Dashboard (BITS Capstone)

## 1. Project Overview
This project builds an end-to-end fraud/anomaly monitoring system for an e-commerce workflow.
It produces curated datasets, a transparent risk score (0–100), an investigation queue, weekly KPIs,
and a 4-page dashboard for executive + ops consumption.

## 2. Business Objective
Reduce fraud-related loss (refund amount + COD RTO proxy) while minimizing false positives and protecting conversion.

### North Star Metric
Monthly fraud loss prevented (₹) = baseline loss - post-control loss

### Guardrail Metrics
- Conversion proxy (orders count / sessions)
- Cancellation rate
- Customer complaint proxy (if not available: track refund disputes / repeated refunds)

## 4. Curated Outputs (Generated by ETL)
### Output 1: fact_orders_enriched.csv
1 row per order with joins + engineered signals + risk score + top reasons.

### Output 2: fact_user_risk_weekly.csv
1 row per user per week with KPIs + avg risk score.

### Output 3: investigation_queue.csv
Ranked queue of orders to investigate with action recommendations.

## 5. ETL Pipeline
### Steps
1) Load + standardize columns & timestamps  
2) Deduplicate tables (documented rules)  
3) Handle missing values (documented strategy)  
4) Join to order grain (1 row per order)  
5) Engineer risk signals (10–15)  
6) Compute risk score (0–100) + banding  
7) Build weekly user summary  
8) Build investigation queue + recommended action

### How to run
- Python: `python etl/etl_pipeline.py`
- Outputs written to `/data/`

## 6. Risk Scoring (Explainable)
Risk score is computed as a weighted sum of normalized signals.
For each order we store:
- risk_score (0–100)
- risk_band (High/Medium/Low)
- top_3_reasons (reason codes + evidence values)
